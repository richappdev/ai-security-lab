Repository Review

Guarded testing architecture

The project is organized around a local API, safety controls, tool modules, lab targets, and generated reports.

Core Areas

  • app/ FastAPI entrypoint, service wiring, and static dashboard pages.
  • safety/ allowlist enforcement, rate limits, policy checks, and audit logging.
  • tools/ passive inspection and low-risk active checks.
  • reports/ Markdown output from scan runs.
  • tests/ unit tests for API, tools, reports, and safety behavior.

Implemented Tools

  • Passive HTTP response header inspection.
  • Passive Cookie attribute inspection.
  • Passive Same-page form discovery.
  • Active low risk Harmless reflected-input marker check.
  • Active low risk One-request HTTP OPTIONS method check.
  • Active low risk One-request route existence check for a known path.
  • Active low risk Two-request security header delta check for a known route.
  • Active low risk GET-only authentication page metadata check for a known route.

Safety Review

Control Location Purpose
Target allowlist targets.allowlist Restricts scans to local lab URLs.
Scope guard safety/scope_guard.py Rejects non-local or non-allowlisted targets before network access.
Policy limits safety/policy.yml Defines timeout and request-rate boundaries.
Audit log safety/audit_log.py Records every tool run with target, status, and result summary.

Review Notes

  • The current API exposes six scan endpoints: passive headers, reflected-input, HTTP methods, route existence, security header delta, and authentication page metadata.
  • Cookie and form tools exist as modules but are not yet exposed as API endpoints.
  • Route existence, header delta, and authentication page metadata checks are exposed in the static UI as fixed-size, allowlist-bound checks.
  • Report generation is Markdown-based and suitable for local audit trails.
